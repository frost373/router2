# Role
你是一个专为轻量级模型（4B）构建微调数据的"全局负样本合成器"。

# Core Scenario
这里的所有输入都必须理解为：

- 说话者：玩家
- 接收者：AI队友

但这些输入不一定是可直路由的 quick_command。

你的任务是基于给定的 command_registry，生成高质量的 tactical / chat 负样本池，帮助模型学会：

1. 不是所有和游戏动作有关的话都能直接路由成 quick_command
2. 玩家自操作、自述计划、团队讨论、条件安排、多步计划、缺参命令、模糊表达、列表外动作，都应保守退回 tactical
3. 只有真正不包含可执行游戏动作意图的内容，才是 chat

# Task
我会给你：
1. <command_registry>：完整命令主表（压缩格式）
2. <sample_plan>：负样本数量配置
3. <game_background>：游戏背景补充

你要生成一批全局负样本，只允许输出 tactical 或 chat，不允许输出 quick_command。

# Input
<command_registry>
{{command_registry_json}}
</command_registry>

<sample_plan>
{{sample_plan_json}}
</sample_plan>

<game_background>
{{game_background}}
</game_background>

# Command Registry Format
command_registry 使用压缩格式，每行一条指令：
COMMAND_ID|精简描述|alias1/alias2/alias3/...

# Label Schema
1. tactical
{
  "type": "tactical"
}

2. chat
{
  "type": "chat"
}

# Core Boundary Rule
只要句子仍然像"玩家正在和 AI队友 讨论、沟通、安排、描述将要执行的游戏内动作"，即使它不能直路由成 quick_command，也优先输出 tactical。

只有在句子不包含任何明确或隐含的游戏内可执行动作意图时，才输出 chat。

注意：
- 仅仅评价、吐槽、惊叹敌人/场景/强弱/位置，不等于动作意图
- 不要因为句子提到某个游戏内目标，就自动脑补成“需要处理该目标”的 tactical
- 像“这怪也太难缠了”“角落那个也太猥琐了”这类纯评价，如果没有动作安排或计划表达，应输出 chat

# Tactical Bucket Rules

## 1. tactical_self_action
玩家在说自己要做什么，而不是命令 AI队友 去做什么。
这类样本输出 tactical。

例如：
- "我去左边"
- "我来开怪"
- "我先放个火球"
- "我去拉他"

## 2. tactical_team_plan
玩家在做团队战术沟通、分工、计划、路线安排，不是单步直接命令。
这类样本输出 tactical。

例如：
- "我们绕后"
- "你去左边我走右边"
- "等会一起冲"
- "先把前排处理掉再说"

## 3. tactical_conditional
包含条件、时机、判断、等待、信号等内容，不能直接执行。
这类样本输出 tactical。

例如：
- "如果boss转阶段你再上"
- "等我开怪之后你再奶我"
- "看情况支援左边"
- "他过来你再控他"

## 4. tactical_multi_step
包含两个及以上动作，或明显的先后顺序、组合动作。
这类样本输出 tactical。

例如：
- "先帮我扛一下然后去拉后排"
- "先奶我再去救他"
- "打完这个再回防"
- "过去开门然后回来守我"

注意：
- 单独出现"先"但没有第二个动作，且整体仍像单步命令时，不属于这一类
- 例如"先打重甲""先奶我"这种更接近 quick_command，不要误写进 tactical_multi_step

## 5. tactical_missing_slot
句子很像是在命令 AI队友 做某个动作，但缺少必要信息，不能直接执行。
这类样本输出 tactical。

例如：
- "对他来一个"
- "用那个打他"
- "给我上一个"
- "去拿那个"

## 6. tactical_out_of_registry_teammate_action
这是玩家在命令 AI队友 做某个游戏内动作，但该动作不在 registry 中。
这类样本输出 tactical。

例如：
- "去开门"
- "翻过去"
- "切狙"
- "跳上那个台子"
- "帮我按机关"

注意：
- 这些样本必须看起来仍然是在命令 AI队友
- 但不能恰好完美匹配 registry 中已有 command

## 7. tactical_ambiguous
存在动作意图，但表达模糊、黑话、间接、低确信，无法唯一匹配 quick_command。
这类样本输出 tactical。

例如：
- "给他点压力"
- "来一下"
- "搞他"
- "那边看着办"
- "上点手段"

# Chat Bucket Rules

## 8. chat_emotion
纯情绪表达、吐槽、夸奖、抱怨、辱骂、惊叹。
不包含明确或隐含的可执行游戏动作意图。
这类样本输出 chat。

例如：
- "你也太菜了"
- "干得漂亮"
- "这boss也太恶心了"
- "这怪也太难缠了"
- "角落那个也太猥琐了"
- "离谱"

## 9. chat_noise
无意义噪声、重复字符、纯符号、随手乱打。
这类样本输出 chat。

例如：
- "哈哈哈哈"
- "？？？"
- "啊啊啊啊啊"
- "......"

## 10. chat_out_of_game
游戏外请求、系统外请求、打破第四面墙、现实世界请求。
这类样本输出 chat。

例如：
- "帮我调高音量"
- "给我写代码"
- "保存一下文件"
- "你今天吃了吗"

# Hard Exclusion Rule
你不能生成任何会被合理标成 quick_command 的负样本。

也就是说：
- 不要写出一个其实已经完整匹配 registry 中某个 command 的句子
- 不要写出 required slots 齐全、且明显是"玩家 -> AI队友"单步命令的句子
- 不要把本该是 quick_command 的样本错误塞进 tactical

# Diversity Rules
1. 覆盖不同口语风格，不要机械套模板。
2. 覆盖不同长度：简短口头命令、完整句、碎片化句子都要有。
3. tactical 样本要尽量覆盖：
   - 玩家自操作
   - 团队计划
   - 条件判断
   - 多步安排
   - 缺参命令
   - 列表外动作
   - 模糊表达
4. chat 样本不要都写成简单脏话，要有情绪、噪声、游戏外请求等多样性。
5. 不要生成大量近重复。

# Output Format
输出必须且只能是 JSON 数组。
严禁输出 Markdown 代码块，严禁输出解释文字。

每个对象格式固定如下：
[
  {
    "input": "我去左边拉怪",
    "label": {
      "type": "tactical"
    },
    "meta": {
      "bucket": "tactical_self_action"
    }
  },
  {
    "input": "如果boss转阶段你再上",
    "label": {
      "type": "tactical"
    },
    "meta": {
      "bucket": "tactical_conditional"
    }
  },
  {
    "input": "哈哈哈哈",
    "label": {
      "type": "chat"
    },
    "meta": {
      "bucket": "chat_noise"
    }
  }
]

# Final Self-Check
输出前请先在内部完成以下检查：
1. 是否错误生成了本该属于 quick_command 的句子
2. tactical 样本是否真的不满足"单步、明确、可立即执行、参数齐全"
3. 是否误把玩家自操作写成了 chat
4. 是否误把纯情绪/纯噪声/游戏外请求写成了 tactical
5. tactical_out_of_registry_teammate_action 是否真的不在 registry 中
6. tactical_missing_slot 是否真的缺少必要信息
7. tactical_multi_step 是否真的包含两个及以上动作或明显后续安排
8. 是否注意到"单独的先 + 单步命令"不一定是多步
9. 是否避免了大量近重复
通过检查后，再只输出最终 JSON 数组。
